# Product Dev Methods

## 方法目标

这个方法层不是额外加一套流程，而是统一回答几个问题：

1. 原型应该先看什么
2. 哪些上下文必须读，哪些不用全量读
3. 文档达到什么程度才算可以进入下一阶段
4. 如何把结果回写为可复查证据

## 最小上下文加载

优先读取顺序：

1. `manifest.json`
2. `00-需求总览.md`
3. `原型解析摘要.md`
4. `需求文档.md`
5. `UI交互设计规范.md`

除非被明确阻塞，不要一次性把整个仓库所有模板和规则全量加载。

## 原型分析四步

1. **识别页面类型**
   - 列表页 / 详情页 / 新增页 / 编辑页 / 弹窗 / Tab 页
2. **识别结构层级**
   - 单表 / 主子表 / 主子孙表
3. **识别交互动作**
   - 查询 / 重置 / 新增 / 编辑 / 查看 / 删除 / 提交 / 保存 / 归档 / 上传附件
4. **识别污染项**
   - 模板残留字段
   - 混入其他业务状态
   - 与当前页面不一致的旧文案

## 产出顺序

1. `原型解析摘要.md`
2. `00-需求总览.md`
3. `需求文档.md`
4. `UI交互设计规范.md`

## Gate 优先

- requirement Gate 通过前，不进入 UI 阶段
- UI Gate 通过前，不沉淀到 `examples/`
- 所有“待确认项”必须显式列出，不能悄悄省略

## 反馈回路

每次真实模拟完成后，要做两件事：

1. 在 `workspace/requests/<request-key>/` 保留完整过程产物
2. 若结果可复用，再沉淀到 `examples/<request-key>/`
