## 阅读约定（必填，建议保留）

### 术语定义

- `搜索筛选`: 表示页面最头部的查询区域。
- `数据展示`: 表示列表、详情、卡片、Tab 等区域中的字段展示。
- `操作按钮`: 表示当前页面所有可以点击的操作入口。
- `按钮展示逻辑`: 表示不同状态、权限、页面模式下按钮是否可见、是否可点。
- `表单定义`: 表示新增、编辑、查看场景下字段的显示、只读、必填、默认值、联动规则。
- `业务规则`: 表示边界条件、校验、联动、删除限制、审批分支等系统和业务约束。
- `系统字段`: 表示系统自动补齐、自动生成、只读回显或隐藏落库的字段。

### 字段来源类型说明

| 类型 | 含义 | 典型场景 | 写法要求 |
|------|------|---------|---------|
| 用户输入 | 用户主动录入或选择 | 文本、数字、日期、下拉 | 写清组件、必填、校验 |
| 系统默认带出 | 根据当前登录人/公司/租户自动回填 | 公司编码、公司名称 | 写清默认来源、是否允许修改 |
| 关联选择自动带出 | 选择 A 后自动带出 B/C | 部门负责人、岗位概述 | 写清触发字段、回填字段、是否只读 |
| 系统计算/生成 | 保存前或展示时系统自动生成 | 单号、拼接字段、统计值 | 不得写成手工输入 |
| 隐藏字段 | 页面不展示但落库或计算需要 | 审批开关、内部标记 | 写清是否前端透传或后端补齐 |
| 查看态只读 | 某些场景可编辑，查看态只读 | 审批状态、系统带出字段 | 写清新增、编辑、查看差异 |

### 文档编写强约束

- 系统已有公共字段、组织字段、人员字段、岗位字段、公司字段，必须写清获取方式，不能只写“数据源”或“用户填写”。
- 如果真实交互是“选择器 + 自动带出编码/名称/负责人”，必须在表单定义中明确写出，不得拆成多个“自由输入字段”。
- 如果真实交互是“弹窗选择器”，必须写清：组件类型、是否弹窗、是否带搜索区、单选/多选、回填字段、展示格式、默认值来源、是否允许改选。
- 如果某字段最终落库，但前端不应传值，必须明确写为“系统生成”或“后端补齐”。
- 如果某字段是展示拼接值，必须标注为“系统计算展示”或“冗余展示字段”。
- 当文档中出现“工号+姓名”“工号和姓名”“人员编码+姓名”等描述时，统一落为显示格式 `工号-姓名`。
- 当文档中出现“公司编码+公司名称”“公司编码和公司名称”等描述时，统一落为显示格式 `公司编码-公司名称`。
- 所有 `申请人` 字段统一使用 `弹窗选择器`，不再写成“默认当前登录人”或“普通输入框”。
- 表单中的所有 `Select` 组件均默认支持搜索；如果不是，必须显式说明原因。
- 如果原始资料中某一块已经是表格，优化后优先保留表格结构，不要退化成口语化段落。
- 需求文档要优先写清“数据来源、展示格式、是否可编辑、何时回写、状态如何流转”，不要只描述页面长什么样。
- 涉及联动链路时，必须写清“来源模块 -> 目标模块 -> 触发时机 -> 写入字段 -> 状态变化”，不能只写“自动同步”。
- 涉及审批流时，至少写清“未接审批流”和“已接审批流”两种分支逻辑。
- 涉及详情页时，必须补充 `详情页按钮行为`，至少写清新增、编辑、查看三种进入方式下的按钮展示差异。
- 涉及主表、子表场景时，必须明确标注 `主表`、`子表`，并写清是否支持多条维护。
- 如果原型 HTML 已明确标记 `主表 / 子表 / 孙表`，需求文档必须原样保留该层级标记，不得合并成一张无层级字段表；如存在多个子表或多个孙表，统一写成 `子表-XXX`、`孙表-XXX`。
- 删除前校验、唯一性校验、编码规则、组织归属、审批流开关等系统级规则，不能只写一句“按系统逻辑处理”，必须显式展开。

### 专项业务链路规则（按需保留）

- 如果当前业务存在明显的链路分段，例如 `线索 -> 跟进 -> 成交`、`项目 -> 成员 -> 任务`、`单据 -> 明细 -> 批次`，建议按链路或层级拆块描述，不要把所有字段堆在一张表中。
- 如果某一业务存在明显的行业专属状态流转，应把“正常路径”和“异常路径”分别写清。
- 如果某一业务依赖外围能力，例如消息、邮件、短信、外部系统同步，应明确写为“本轮范围内”还是“预留能力”。
- 如果这部分与当前需求无关，应直接删除，不要保留成无关说明。
