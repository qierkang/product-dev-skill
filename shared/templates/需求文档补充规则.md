# Product Dev 需求文档补充规则

## 全局展示口径

- `工号+姓名`、`人员编码+姓名`、`账号+姓名` 统一写为 `工号-姓名`
- `公司编码+公司名称` 统一写为 `公司编码-公司名称`
- `部门编码+部门名称` 统一写为 `部门编码-部门名称`
- `岗位编码+岗位名称` 统一写为 `岗位编码-岗位名称`

## 头部公共说明块

以下内容建议直接保留在 `需求文档模板.md` 顶部，而不是只写在补充说明里：

- `术语定义`
- `字段来源类型说明`
- `文档编写强约束`
- `通用系统字段快捷块`
- `系统字段交互规格模板`

## 全局交互规则

- 所有 `申请人` 字段统一使用弹窗选择器
- 所有 `Select` 默认支持搜索
- 只要是选择器、下拉框、TreeSelect、弹窗选择器，都应考虑大数据量场景并写明是否支持搜索

## 写作优先级

1. 先写清业务链路和模块关系
2. 再写清页面区块、字段来源和按钮行为
3. 再写清状态流转、联动回写、边界规则
4. 最后再补待确认项

## 结构化表达规则

- 优先使用表格描述字段、按钮、状态和接口
- 主表、子表、明细表必须拆块写，不要混在一个段落里
- 内部公共接口、系统回填能力、编码规则能力必须单列说明

## 字段描述规则

- 字段描述必须包含：来源、展示格式、是否可编辑、是否必填、何时回写
- 凡是系统字段、组织字段、人员字段、岗位字段，优先说明来源接口或来源能力
- 只读字段要明确“前端只读展示”还是“后端补齐”

## 联动与状态规则

- 涉及联动时，必须写清：来源模块 → 目标模块 → 触发时机 → 写入字段 → 状态变化
- 涉及审批流时，必须区分未接审批流和已接审批流
- 涉及删除、作废、撤回、提交时，必须写清前置条件和失败提示

## 推荐使用方式

1. 以 `需求文档模板.md` 为主模板生成正文
2. 生成前先对照本文件做一次头部公共说明块校验
3. 如需求涉及系统字段、组织字段、人员字段、岗位字段，优先复用模板中的快捷块和交互规格表

## 专项链路规则

- 如果需求存在明显的主链路或层级结构，建议把每一段业务拆开写清，而不是笼统描述。
- 如果某一业务是行业专属模板，请在生成后主动删除与当前需求无关的专项说明。
